## 四、Phase 3 ── 精準度修復

### 3a. ARLPLU71 保誠新美滿相傳 ❌ +672% → ✅ 應對齊

**根本原因**:Manifest 條目 `period: 1`(躉繳),但 JSON 內 `plan_year: 6`(6 年期)。前端按 6 年期顯示卻按 1 年期算保額,造成 SA 算高 7 倍。

**修法**:Manifest 條目 `period` 改 1 → 6,與 JSON `plan_year=6` 對齊。

### 3b. NUIW6602 / NUPW0202 ⚠️ 標警告

**根本原因**(不是引擎錯,是源 Excel 沒填):
- `NUIW6602` JSON `base_sa=12,000`(太小),Drew 基準 SA 應為 ~50,000~100,000 USD
- `NUPW0202` JSON `base_sa=9,999,999`(明顯是 Excel 預設極大值)、`base_premium=1,350,375`(同樣異常)

**處理**:在 manifest 標 `needs_revalidation: true` + `revalidation_note`,UI 可以後續加 ⚠️ icon 提醒。

**待用戶補**:打開兩個商品的原始 Excel,用「41 歲男 + USD 50,000」基準重抽 JSON。

### 3c. 保誠 ARLPLU30 / 57 / 64 ⚠️ -5~-8% 偏低

**根本原因**:`prudential_v2` 引擎可能漏算某個欄位(`ARB_PV` 或 `TBS_CV` 累積項)。

**處理**:暫時保留(誤差在「可接受 < 10%」邊緣),等之後重抽或調引擎再修。

### 3d. UPD061 載入失敗 ✅ 修好

**根本原因**:Manifest 條目 `path: "UPD061.json"`(指向頂層),但實際檔案在 `data/skl/UPD061.json`。

**修法**:Path 改成 `skl/UPD061.json`。**那個討厭的 alert 從此不會再跳**。

### 3e. FNW Period 不一致 ✅ 修好

JSON 內 `period=2`,Manifest 寫 `period=1`。改 Manifest 為 2。

### 3f. 刪 3 個 Ghost Entry ✅

`FKD_2Y` / `FMW_2Y` / `FNW_2Y` — manifest 有條目但實體檔不存在。**直接刪除**(原本就是壞掉的引用)。
